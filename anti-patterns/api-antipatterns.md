# REST API Anti-Patterns: No Pagination, Returning Entities, and Missing Rate Limiting

## When to Use
- Designing REST APIs
- Reviewing API code generated by small LLMs
- API code review checklist
- Understanding REST best practices

## Standard Pattern

```markdown
# WRONG: No pagination (returns all records)
GET /api/users
# Returns 100,000 users — crashes the client!

# CORRECT: Always paginate
GET /api/users?page=1&per_page=20
# Response: { "data": [...], "total": 100000, "page": 1, "per_page": 20 }

# WRONG: Returning database entities directly
# return jsonify(user)  # Exposes password_hash, internal fields!

# CORRECT: Return DTOs (Data Transfer Objects)
# return jsonify(UserDto.from_entity(user))  # Only expose safe fields

# WRONG: No rate limiting
# Anyone can make unlimited requests — DDoS vulnerability!

# CORRECT: Implement rate limiting
# 429 Too Many Requests with Retry-After header
# Use token bucket or sliding window algorithm

# WRONG: Ignoring CORS
# Frontend on different domain can't call the API!

# CORRECT: Configure CORS properly
# Access-Control-Allow-Origin: https://yourdomain.com
# Access-Control-Allow-Methods: GET, POST, PUT, DELETE
# Access-Control-Allow-Headers: Authorization, Content-Type

# WRONG: Versioning in URL (breaks URLs on version change)
GET /api/v1/users
# When v2 comes, all client URLs break!

# CORRECT: Version in header or use backward-compatible changes
# Accept: application/vnd.myapp.v1+json
# Or: /api/users with backward-compatible changes

# WRONG: No HATEOAS (clients hardcode URLs)
{ "id": 1, "name": "Alice" }
# Client must know URL structure for related resources!

# CORRECT: Include links to related resources (HATEOAS)
{
  "id": 1,
  "name": "Alice",
  "_links": {
    "self": { "href": "/api/users/1" },
    "orders": { "href": "/api/users/1/orders" },
    "profile": { "href": "/api/users/1/profile" }
  }
}

# WRONG: Using verbs in URLs
GET /api/getUsers
POST /api/createUser
DELETE /api/deleteUser/1

# CORRECT: Use nouns (resources) with HTTP methods
GET    /api/users       # List users
POST   /api/users       # Create user
GET    /api/users/1     # Get user
PUT    /api/users/1     # Update user
DELETE /api/users/1     # Delete user

# WRONG: Inconsistent error format
# Sometimes: { "error": "Not found" }
# Sometimes: { "message": "User not found" }
# Sometimes: { "code": 404, "msg": "Not found" }

# CORRECT: Consistent error format
{
  "error": {
    "code": "NOT_FOUND",
    "message": "User with ID 1 not found",
    "details": { "resource": "User", "id": 1 }
  }
}

# WRONG: Returning 200 OK for errors
HTTP/1.1 200 OK
{ "error": "Invalid email" }

# CORRECT: Use proper HTTP status codes
HTTP/1.1 400 Bad Request
{ "error": { "code": "VALIDATION_ERROR", "field": "email" } }
```

## Common Mistakes
- No pagination — returns all records, crashes clients with large datasets
- Returning database entities directly — exposes internal fields like password_hash
- No rate limiting — DDoS vulnerability, anyone can make unlimited requests
- Ignoring CORS — frontend on different domain cannot call the API
- Using verbs in URLs — should use nouns (resources) with HTTP methods
- Inconsistent error format — different endpoints return different error shapes
- Returning 200 OK for errors — should use proper HTTP status codes (400, 404, 500)
- No versioning strategy — breaking changes force all clients to update simultaneously

## Gotchas
- Pagination is NOT optional for any endpoint that returns a list. Always paginate.
- DTOs prevent over-fetching and protect sensitive fields. Never return entities directly.
- Rate limiting should return 429 with Retry-After header.
- CORS preflight (OPTIONS) must be handled for cross-origin requests.
- HTTP status codes matter: 200 (OK), 201 (Created), 204 (No Content), 400 (Bad Request), 401 (Unauthorized), 403 (Forbidden), 404 (Not Found), 409 (Conflict), 422 (Unprocessable), 429 (Too Many Requests), 500 (Server Error).
- HATEOAS makes APIs self-discoverable. Clients follow links instead of constructing URLs.
- API versioning: URL versioning (/v1/) is simple but breaks URLs. Header versioning is cleaner.
- Consistent error format across all endpoints makes client error handling easier.
