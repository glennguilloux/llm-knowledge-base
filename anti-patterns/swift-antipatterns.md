# Swift Anti-Patterns: Force Unwrapping, Massive View Controllers, and Outdated Patterns

## When to Use
- Reviewing Swift code generated by small LLMs
- Training LLMs to write idiomatic Swift instead of Objective-C-in-Swift
- Code review checklist for Swift projects
- Understanding why certain Swift patterns are preferred

## Standard Pattern

```swift
// WRONG: Force unwrapping everywhere
func processUser(_ user: User?) {
    let name = user!.name       // Crashes if nil
    let email = user!.email     // Crashes if nil
    let city = user!.address!.city  // Double crash risk!
}

// CORRECT: Use safe unwrapping with guard or if let
func processUser(_ user: User?) {
    guard let user = user else { return }
    let name = user.name
    let email = user.email
    let city = user.address?.city ?? "Unknown"
}

// WRONG: Not using guard — pyramid of doom
func validateAndProcess(name: String?, email: String?, age: Int?) {
    if let name = name {
        if !name.isEmpty {
            if let email = email {
                if email.contains("@") {
                    if let age = age {
                        if age >= 0 {
                            print("Processing \(name)")
                        }
                    }
                }
            }
        }
    }
}

// CORRECT: Use guard for clean early returns
func validateAndProcess(name: String?, email: String?, age: Int?) {
    guard let name = name, !name.isEmpty else { return }
    guard let email = email, email.contains("@") else { return }
    guard let age = age, age >= 0 else { return }
    print("Processing \(name)")
}

// WRONG: Objective-C patterns in Swift (using NSString, NSNumber, etc.)
let name: NSString = NSString(string: "Alice")
let number: NSNumber = NSNumber(value: 42)
let array: NSMutableArray = NSMutableArray()

// CORRECT: Use Swift native types
let name: String = "Alice"
let number: Int = 42
var array: [Any] = []

// WRONG: Massive view controller (UIKit)
class ProfileViewController: UIViewController {
    // 2000 lines of code: networking, database, UI, business logic, etc.
    @IBOutlet weak var nameLabel: UILabel!
    @IBOutlet weak var emailLabel: UILabel!
    @IBOutlet weak var tableView: UITableView!
    
    var users: [User] = []
    var isLoading = false
    
    // Network calls, database queries, table view data source, etc.
    // All in one class!
}

// CORRECT: Separate concerns with MVVM
class ProfileViewModel: ObservableObject {
    @Published var user: User?
    @Published var isLoading = false
    
    func loadUser() async {
        isLoading = true
        user = try? await userService.fetchCurrentUser()
        isLoading = false
    }
}

struct ProfileView: View {
    @StateObject var viewModel = ProfileViewModel()
    
    var body: some View {
        VStack {
            if viewModel.isLoading {
                ProgressView()
            } else if let user = viewModel.user {
                Text(user.name)
                Text(user.email)
            }
        }
        .task { await viewModel.loadUser() }
    }
}

// WRONG: Not using enums for states
var isLoggedIn: Bool = false
var hasError: Bool = false
var errorMessage: String? = nil
// These can be in inconsistent states!

// CORRECT: Use enum for mutually exclusive states
enum AuthState {
    case loggedOut
    case loading
    case loggedIn(User)
    case error(String)
}

var authState: AuthState = .loggedOut
// Compiler ensures only one state at a time

// WRONG: Ignoring Sendable warnings in concurrent code
class Counter {
    var value = 0  // Not thread-safe!
    func increment() { value += 1 }
}

// CORRECT: Use actor for thread-safe mutable state
actor Counter {
    private var value = 0
    func increment() { value += 1 }
    func getValue() -> Int { value }
}

// WRONG: Using class when struct should be used
class Point {
    var x: Double
    var y: Double
    init(x: Double, y: Double) { self.x = x; self.y = y }
}
// Reference semantics — shared mutable state!

// CORRECT: Use struct for value types
struct Point {
    var x: Double
    var y: Double
}
// Value semantics — copied on assignment

// WRONG: Not using Result type for error handling
func fetchUser(completion: @escaping (User?, Error?) -> Void) {
    // Caller must check both user and error — easy to forget!
}

// CORRECT: Use Result for type-safe error handling
func fetchUser(completion: @escaping (Result<User, Error>) -> Void) {
    // Compiler enforces handling both cases
}
```

## Common Mistakes
- Force unwrapping everywhere — crashes if value is nil, use guard/if let
- Not using guard — pyramid of doom from nested if let statements
- Objective-C patterns in Swift — NSString, NSNumber instead of String, Int
- Massive view controllers — violates single responsibility principle
- Not using enums for states — boolean flags instead of exhaustive enum states
- Ignoring Sendable warnings — data races in concurrent code
- Using class when struct should be used — value semantics prevent shared mutable state

## Gotchas
- `!` (force unwrap) should be used extremely rarely. If the value is nil, your app crashes.
- `guard` is preferred over `if let` when you want the unwrapped value for the rest of the scope.
- Swift structs are value types (copied). Classes are reference types (shared). Default to struct.
- Enums with associated values are perfect for state machines. The compiler ensures exhaustive handling.
- Actors protect shared mutable state. Use them instead of locks or dispatch queues.
- `@MainActor` ensures UI updates happen on the main thread. Use it for view models.
- `Result<Success, Failure>` is type-safe error handling for callbacks.
- Small LLMs often generate Objective-C-style Swift because their training data includes older code.
- `Sendable` conformance is required for types shared across concurrency domains.
