# PHP Anti-Patterns: Deprecated Functions and Dangerous Practices

## When to Use
- Reviewing PHP code generated by small LLMs
- Migrating legacy PHP code to modern PHP 8.x
- Training LLMs to avoid deprecated PHP patterns
- Code review checklist for PHP projects

## Standard Pattern

```php
<?php

// WRONG: Using mysql_ functions (removed in PHP 7.0!)
$conn = mysql_connect('localhost', 'user', 'pass');
mysql_select_db('mydb', $conn);
$result = mysql_query('SELECT * FROM users');
while ($row = mysql_fetch_assoc($result)) {
    echo $row['name'];
}

// CORRECT: Use PDO with prepared statements
$pdo = new PDO('mysql:host=localhost;dbname=mydb', 'user', 'pass', [
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
]);
$stmt = $pdo->query('SELECT * FROM users');
while ($row = $stmt->fetch(PDO::FETCH_ASSOC)) {
    echo htmlspecialchars($row['name'], ENT_QUOTES, 'UTF-8');
}

// WRONG: Not using prepared statements (SQL injection)
$email = $_GET['email'];
$result = $pdo->query("SELECT * FROM users WHERE email = '$email'");

// CORRECT: Use prepared statements
$stmt = $pdo->prepare('SELECT * FROM users WHERE email = :email');
$stmt->execute(['email' => $_GET['email']]);

// WRONG: Global variables everywhere
global $db;
global $config;
global $user;
function doSomething() {
    global $db;
    $db->query('...');
}

// CORRECT: Use dependency injection
class UserService {
    public function __construct(
        private PDO $db,
        private array $config,
    ) {}
    
    public function doSomething(): void {
        $this->db->query('...');
    }
}

// WRONG: Mixed PHP and HTML (spaghetti code)
// <html><body>
// <?php $users = $db->query('SELECT * FROM users'); ?>
// <?php while ($user = $users->fetch()): ?>
//     <div><?php echo $user['name']; ?></div>
// <?php endwhile; ?>
// </body></html>

// CORRECT: Separate logic from presentation
// Controller:
$users = $userService->getAllUsers();
require 'templates/users.php';

// Template (users.php):
// <?php foreach ($users as $user): ?>
//     <div><?= htmlspecialchars($user->name, ENT_QUOTES, 'UTF-8') ?></div>
// <?php endforeach; ?>

// WRONG: Using md5/sha1 for password hashing
$hash = md5($password);   // Crackable in seconds
$hash = sha1($password);  // Crackable in seconds

// CORRECT: Use password_hash with ARGON2ID
$hash = password_hash($password, PASSWORD_ARGON2ID);
if (password_verify($inputPassword, $storedHash)) {
    // Login successful
}

// WRONG: Suppressing errors with @
$result = @file_get_contents('https://api.example.com/data');
// If it fails, you'll never know why!

// CORRECT: Handle errors properly
try {
    $result = file_get_contents('https://api.example.com/data');
    if ($result === false) {
        throw new RuntimeException('API request failed');
    }
} catch (Exception $e) {
    error_log('API error: ' . $e->getMessage());
    // Handle gracefully
}

// WRONG: Not using Composer autoloading
require_once 'lib/Database.php';
require_once 'lib/User.php';
require_once 'lib/UserService.php';
// Manual requires are error-prone and don't scale

// CORRECT: Use Composer autoloading
// composer.json: { "autoload": { "psr-4": { "App\\": "src/" } } }
require_once 'vendor/autoload.php';
// All classes autoloaded automatically

// WRONG: Using die() for error handling
if (!$db) {
    die('Database connection failed');
}

// CORRECT: Throw exceptions
if (!$db) {
    throw new RuntimeException('Database connection failed');
}

// WRONG: Not validating input
$email = $_POST['email'];
// Use $email directly in queries, emails, etc.

// CORRECT: Validate and sanitize
$email = filter_input(INPUT_POST, 'email', FILTER_VALIDATE_EMAIL);
if ($email === false) {
    throw new InvalidArgumentException('Invalid email');
}
```

## Common Mistakes
- Using mysql_* functions — removed in PHP 7.0, use PDO or MySQLi
- Not using prepared statements — SQL injection vulnerability
- Global variables — makes code untestable, use dependency injection
- Mixed PHP and HTML — spaghetti code, use templates
- md5/sha1 for passwords — trivially crackable, use password_hash()
- Suppressing errors with @ — hides bugs, makes debugging impossible
- Not using Composer — manual dependency management, no autoloading
- Not validating input — XSS, SQL injection, and other vulnerabilities

## Gotchas
- `mysql_*` functions were removed in PHP 7.0. If you see them, the code is at least 8 years outdated.
- `password_hash()` with `PASSWORD_ARGON2ID` is the strongest option. `PASSWORD_BCRYPT` is the fallback.
- `md5()` and `sha1()` are NOT password hashes — they're general-purpose hashes that are trivially crackable.
- The `@` error suppression operator hides ALL errors, including fatal ones. Never use it in production.
- `global` variables make code untestable and create hidden dependencies. Use DI instead.
- Composer is the standard PHP dependency manager. Not using it means manual dependency management.
- `filter_input()` and `filter_var()` are the proper way to validate input in PHP.
- Mixed PHP/HTML is a maintenance nightmare. Use a template engine (Twig, Blade) or separate concerns.
- `htmlspecialchars()` with `ENT_QUOTES` and `UTF-8` is the minimum for XSS prevention.
