# Kotlin Anti-Patterns: Common Mistakes Small LLMs Make

## When to Use
- Reviewing Kotlin code generated by small LLMs
- Training LLMs to write idiomatic Kotlin instead of Java-in-Kotlin
- Code review checklist for Kotlin projects
- Understanding why certain Kotlin patterns are preferred

## Standard Pattern

```kotlin
// WRONG: Overusing !! (not-null assertion) — defeats null safety
fun processUser(user: User?) {
    val name = user!!.name  // Crashes if user is null
    val email = user!!.email
}

// CORRECT: Use safe calls and scope functions
fun processUser(user: User?) {
    user?.let {
        val name = it.name
        val email = it.email
        // Process...
    }
}

// WRONG: Java-style code in Kotlin (not using Kotlin features)
fun getActiveUsers(users: List<User>): List<User> {
    val result = ArrayList<User>()
    for (i in 0 until users.size) {
        val user = users[i]
        if (user.isActive) {
            result.add(user)
        }
    }
    return result.sortedBy { it.name }
}

// CORRECT: Use Kotlin collection operations
fun getActiveUsers(users: List<User>): List<User> {
    return users.filter { it.isActive }.sortedBy { it.name }
}

// WRONG: Ignoring scope functions — verbose null handling
fun sendEmail(user: User?) {
    if (user != null) {
        if (user.email != null) {
            if (user.email.isNotEmpty()) {
                emailService.send(user.email, "Hello!")
            }
        }
    }
}

// CORRECT: Use scope functions for clean null handling
fun sendEmail(user: User?) {
    user?.email?.takeIf { it.isNotEmpty() }?.let { email ->
        emailService.send(email, "Hello!")
    }
}

// WRONG: Mutable collections everywhere
val users = mutableListOf<User>()
val scores = mutableMapOf<String, Int>()
val uniqueIds = mutableSetOf<Int>()

// CORRECT: Use immutable collections by default
val users = listOf<User>()
val scores = mapOf<String, Int>()
val uniqueIds = setOf<Int>()
// Only use mutable when you actually need to modify

// WRONG: Not using data class for DTOs
class UserDto(val id: Int, val name: String, val email: String) {
    override fun equals(other: Any?): Boolean { /* manual */ }
    override fun hashCode(): Int { /* manual */ }
    override fun toString(): String { /* manual */ }
}

// CORRECT: Use data class — auto-generates equals, hashCode, toString, copy
data class UserDto(val id: Int, val name: String, val email: String)

// WRONG: Premature sealed class usage (over-engineering)
sealed class StringWrapper
data class NonEmptyString(val value: String) : StringWrapper()
object EmptyString : StringWrapper()

// CORRECT: Use sealed class only when you have a genuine closed hierarchy
sealed class PaymentResult {
    data class Success(val transactionId: String) : PaymentResult()
    data class Failure(val reason: String) : PaymentResult()
    object Pending : PaymentResult()
}

// WRONG: Over-abstracting with interfaces for everything
interface UserRepository { fun findById(id: Int): User? }
class UserRepositoryImpl : UserRepository { /* ... */ }

// CORRECT: Start with concrete class, introduce interface only when needed
class UserRepository { fun findById(id: Int): User? = null }

// WRONG: Using var for everything (Java habit)
var name = "Alice"
var age = 30
var email = "alice@example.com"

// CORRECT: Use val by default, var only when mutation is needed
val name = "Alice"
val age = 30
val email = "alice@example.com"

// WRONG: Not using extension functions
fun capitalizeWords(text: String): String {
    return text.split(" ").joinToString(" ") { it.replaceFirstChar { c -> c.uppercase() } }
}

// CORRECT: Use extension functions to add behavior to existing types
fun String.capitalizeWords(): String =
    split(" ").joinToString(" ") { it.replaceFirstChar { c -> c.uppercase() } }
```

## Common Mistakes
- Overusing !! (not-null assertion) — defeats Kotlin's null safety, causes NPE crashes
- Java-style code in Kotlin — for loops, mutable everything, not using Kotlin idioms
- Ignoring scope functions — let, also, run, apply make null handling elegant
- Mutable collections everywhere — default to immutable (listOf, mapOf, setOf)
- Not using data class — manual equals/hashCode/toString instead of auto-generated
- Premature sealed class usage — over-engineering simple value wrappers
- Over-abstracting with interfaces — start with concrete class, introduce interface when needed

## Gotchas
- `!!` should be used extremely rarely. If you find yourself using it more than once in a function, rethink your approach.
- Small LLMs often generate Java-style Kotlin because their training data includes more Java than Kotlin.
- `val` vs `var`: default to `val`. The compiler will tell you when you need `var`.
- Immutable collections (`listOf`, `mapOf`, `setOf`) should be the default. Use `mutableListOf` only when you need to add/remove elements.
- `data class` is not just syntactic sugar — it gives you `copy()` for immutable updates, which is essential for state management.
- Scope functions are not interchangeable. `let` transforms, `also` does side effects, `run` executes and returns, `apply` configures and returns receiver.
- Extension functions are resolved statically — they don't modify the class. Use them for utility functions, not for polymorphic behavior.
- `sealed class` is powerful but should model genuine closed hierarchies (state machines, result types), not simple value wrappers.
